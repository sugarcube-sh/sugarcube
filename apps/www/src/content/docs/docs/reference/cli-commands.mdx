## `init`

Initialize a new sugarcube project interactively.

### Usage

```bash frame="none"
npx @sugarcube-sh/cli init [options]
```

### Description

The `init` command walks you through setting up sugarcube in your project. It:

- Detects existing design tokens, or prompts you to choose a starter kit
- Optionally adds CUBE CSS files
- Optionally adds components from the registry
- Optionally installs the Vite plugin
- Installs `@sugarcube-sh/cli` as a dev dependency

No config file is created — the project works with auto-discovery and sensible defaults. See [Configuration](/docs/configuration) for customizing settings later.

### Options

<Table>
| Option | Description | Default |
|--------|-------------|---------|
| `--tokens <dir>` | Design tokens directory | Auto-detected based on `src/` |
| `--cube <dir>` | CUBE CSS output directory | Auto-detected based on `src/` |
| `--components <dir>` | Components output directory | Auto-detected based on `src/` |
</Table>

<SiteAside type="note">
    Requires an interactive terminal. For CI or non-interactive use, use the individual commands (`generate`, `cube`, `components`) directly.
</SiteAside>

### Examples

```bash frame="none"
# Interactive initialization with defaults
npx @sugarcube-sh/cli init

# Specify custom directories
npx @sugarcube-sh/cli init --tokens ./tokens --cube ./css
```

## `generate`

Generate CSS variables and utility classes from your design tokens.

### Usage

```bash frame="none"
npx sugarcube generate [options]
```

### Description

The `generate` command processes your design tokens and generates:

- CSS variables from your tokens
- Utility classes based on your configuration
- Outputs CSS files to your configured styles directory

The command supports **zero-config** usage with automatic resolver discovery, or you can customize everything via CLI flags.

<SiteAside type="note">
    This command is not needed if you're using the **Vite plugin**, which generates virtual CSS modules automatically.
</SiteAside>

### Options

<Table>
| Option | Description | Default |
|--------|-------------|---------|
| `--resolver <path>` | Path to token resolver file | Auto-discovered recursively |
| `--variables <path>` | Output path for CSS variables | `src/styles/variables.gen.css` |
| `--utilities <path>` | Output path for utility classes | `src/styles/utilities.gen.css` |
| `--fluid-min <number>` | Min viewport for fluid scaling | `320` |
| `--fluid-max <number>` | Max viewport for fluid scaling | `1200` |
| `--color-fallback <strategy>` | Color fallback: `native` or `polyfill` | `native` |
| `--input <modifier=value>` | Select a modifier context for this build (repeatable) | - |
| `--selector <selector>` | CSS selector for `--input` output | `:root` |
| `--variables-only` | Generate only CSS variables, skip utilities | - |
| `--utilities-only` | Generate only utilities, skip CSS variables | - |
| `--force` | Skip overwrite confirmation | - |
| `-s, --silent` | Suppress logs and prompts | - |
| `-w, --watch` | Watch for changes and regenerate automatically | - |
</Table>

### Examples

```bash frame="none"
# Zero-config - auto-discovers resolver, uses defaults
npx sugarcube generate

# Specify resolver explicitly
npx sugarcube generate --resolver ./tokens.resolver.json

# Custom output paths
npx sugarcube generate --variables dist/tokens.css --utilities dist/utilities.css

# Override fluid viewport range
npx sugarcube generate --fluid-min 375 --fluid-max 1440

# Use polyfill for wide gamut colors
npx sugarcube generate --color-fallback polyfill

# Generate per-brand CSS files (each at :root)
npx sugarcube generate --input brand=ocean --variables dist/ocean.css --variables-only
npx sugarcube generate --input brand=forest --variables dist/forest.css --variables-only
npx sugarcube generate --utilities-only

# Generate with a custom selector
npx sugarcube generate --input theme=dark --selector "[data-theme='dark']" --variables dist/dark.css

# Override config file values temporarily
npx sugarcube generate --fluid-max 1600

# Watch mode - regenerate on token or markup file changes
npx sugarcube generate --watch
```

### Watch Mode

The `--watch` flag enables continuous regeneration during development:

- Watches your token files (resolver and all referenced JSON files)
- Watches markup files (`.tsx`, `.jsx`, `.vue`, `.svelte`, `.astro`, `.html`, etc.) for utility class changes
- Regenerates CSS when changes are detected
- Shows timing information for each regeneration

```bash frame="none"
npx sugarcube generate --watch
# [sugarcube] Generated in 42ms
# [sugarcube] Watching 5 token files + markup files...
# [sugarcube] colors.json → src/styles/tokens.css (38ms)
```

Press `Ctrl+C` to stop watching.

<SiteAside type="caution">
    Changes to the config file are not watched. You must restart the command to apply new settings.
</SiteAside>

### Flag Behavior

**Config File Priority:**
1. If a config file exists, it's loaded first
2. CLI flags override config values for that run only
3. Config file is never modified by flags

**Without Config File:**
- Auto-discovers `*.resolver.json` files recursively
- All flags use default values unless specified
- Searches entire project tree (excluding build folders)

**Example - Testing Different Values:**
```bash frame="none"
# Your config has fluid.max = 1200
# Test with a larger viewport without changing config:
npx sugarcube generate --fluid-max 1600
# Config file still has 1200 - only this run used 1600
```

### Requirements

**One of:**
- A sugarcube configuration file, OR
- A `*.resolver.json` file in your project (auto-discovered), OR
- The `--resolver` flag with a valid resolver file path

If none are found, the command reports that no design tokens were found and suggests running `npx @sugarcube-sh/cli init` to set up your project.

### Output

The command generates CSS files at your configured paths:

- CSS variables file (default: `src/styles/variables.gen.css`)
- Utility classes file (default: `src/styles/utilities.gen.css`)

## `validate`

Validate design token files for W3C compliance and correctness.

### Usage

```bash frame="none"
npx sugarcube validate [paths...]
```

### Description

The `validate` command checks your design token files for:

- W3C Design Tokens Format compliance
- Correct JSON syntax
- Token reference resolution
- Consistency across token files

### Arguments

<Table>
| Argument | Description |
|----------|-------------|
| `[paths...]` | Optional token files or directories to validate (e.g., `src/design-tokens`) |
</Table>

### Examples

```bash frame="none"
# Validate tokens using config file
npx sugarcube validate

# Validate specific files or directories
npx sugarcube validate src/design-tokens
npx sugarcube validate tokens/colors.json tokens/spacing.json

# Validate multiple paths
npx sugarcube validate src/tokens/colors src/tokens/spacing
```

### Behavior

- **With no paths**: Tries to load config/resolver (including auto-discovery) and validates those tokens
- **With paths**: Validates only the specified files or directories using default transform settings
- **No config, no paths, no resolver found**: Error - paths must be specified

## `components`

Install copy-to-use components for your framework.

### Usage

```bash frame="none"
npx sugarcube components [components...] [options]
```

### Description

The `components` command allows you to browse and install pre-built components from the sugarcube component registry. Components are installed as copy-to-use files in your project.

### Arguments

<Table>
| Argument | Description |
|----------|-------------|
| `[components...]` | Optional component names to install (e.g., `button card`) |
</Table>

### Options

<Table>
| Option | Description |
|--------|-------------|
| `-f, --framework <type>` | Framework to use (`react`, `css-only`) |
| `--output <dir>` | Components output directory (auto-detected based on `src/`, or from config) |
| `-s, --silent` | Suppress logs and prompts |
| `-o, --overwrite` | Overwrite existing files without confirmation |
</Table>

### Examples

```bash frame="none" 
# Interactive mode - browse and select components
npx sugarcube components

# Install specific components
npx sugarcube components button card --framework react

# Install CSS-only components
npx sugarcube components button --framework css-only

# Specify custom output directory
npx sugarcube components button --output src/ui

# Overwrite existing files
npx sugarcube components button --overwrite
```

### Behavior

- **Interactive mode**: If no components or framework are specified, you'll be prompted to select a framework and browse available components
- **Non-interactive mode**: Requires `--framework` flag when specifying components
- **Dependencies**: Automatically installs required npm dependencies for React components
- **Overwrite warnings**: Shows warnings before overwriting existing files (unless `--overwrite` is used)
- **Works without a config file**: Defaults to `src/components/ui` or `components/ui` based on whether your project has a `src/` directory. Use `--output` to override.
- **Does not modify config files** - flag values are temporary for the command run

### Supported Frameworks

- `react` - React components with TypeScript
- `css-only` - Pure CSS components

## `cube`

Add CUBE CSS architecture to your project.

### Usage

```bash frame="none"
npx sugarcube cube [options]
```

### Description

The `cube` command installs the CUBE CSS methodology files into your project. CUBE CSS (Composition, Utility, Block, Exception) provides a structured approach to writing CSS.

### Options

<Table>
| Option | Description |
|--------|-------------|
| `-s, --silent` | Suppress logs and prompts |
| `-f, --force` | Skip overwrite confirmation |
| `-o, --output <dir>` | CUBE CSS output directory (auto-detected based on `src/`, or from config) |
</Table>

### Examples

```bash frame="none"
# Add CUBE CSS files
npx sugarcube cube

# Specify custom output directory
npx sugarcube cube -o src/styles/cube

# Skip overwrite confirmation
npx sugarcube cube --force

# Silent mode
npx sugarcube cube --silent
```

### Output

The command creates CUBE CSS directory structure in your configured styles directory:

```
<styles-dir>/
├── blocks/
├── compositions/
├── global/
└── utilities/
```

Each directory includes starter files. These are starting points that you can modify, add to, or remove as needed.

### Behavior

- Creates the output directory if it doesn't exist
- Shows warnings before overwriting existing files (unless `--force` is used)
- **Works without a config file**: Defaults to `src/styles` or `styles` based on whether your project has a `src/` directory. Use `-o` to override.
- **Does not modify config files** - flag values are temporary for the command run

## Command Workflow

A typical sugarcube workflow:

1. **Initialize** your project:
   ```bash frame="none"
   npx @sugarcube-sh/cli init
   ```

2. **Generate CSS** (if not using Vite plugin):
   ```bash frame="none"
   npx sugarcube generate

   # Or use watch mode during development
   npx sugarcube generate --watch
   ```

3. **Add components** as needed:
   ```bash frame="none"
   npx sugarcube components
   ```

4. **Add CUBE CSS** (optional):
   ```bash frame="none"
   npx sugarcube cube
   ```

5. **Validate tokens** during development:
   ```bash frame="none"
   npx sugarcube validate
   ```
