Sugarcube supports theming through **modifiers** — named variations that override your base token values.

## Basic: zero-config dark mode

Define a modifier in your [resolver document](/docs/resolver) and you're done. No config file needed.

```json title="tokens.resolver.json"
{
  "version": "2025.10",
  "resolutionOrder": [
    {
      "type": "set",
      "name": "base",
      "sources": [
        { "$ref": "./colors.json" },
        { "$ref": "./spacing.json" }
      ]
    },
    {
      "type": "modifier",
      "name": "theme",
      "default": "light",
      "contexts": {
        "light": [],
        "dark": [{ "$ref": "./dark.json" }]
      }
    }
  ]
}
```

Sugarcube generates:

```css
:root {
  --color-surface: #ffffff;
  --color-text: #1a1a1a;
}

[data-theme="dark"] {
  --color-surface: #1a1a1a;
  --color-text: #ffffff;
}
```

The default context ("light") goes to `:root`. Non-default contexts get a `[data-{modifier}="{context}"]` selector automatically.

### Using it in HTML

Set the data attribute to switch themes:

```html
<!-- Light mode (default — no attribute needed) -->
<html>

<!-- Dark mode -->
<html data-theme="dark">
```

You can scope themes to specific elements too:

```html
<div data-theme="dark">
  <!-- only this section uses dark variables -->
</div>
```

### Override files

Override files only need the tokens that change. Everything else inherits from the base set:

```json title="dark.json"
{
  "color": {
    "$type": "color",
    "surface": { "$value": "#1a1a1a" },
    "text": { "$value": "#ffffff" }
  }
}
```

<SiteAside type="tip">
  Keep override files focused. Only include tokens that actually change — this makes it clear what each theme modifies.
</SiteAside>

### More than two contexts

You can define as many contexts as you need:

```json
{
  "type": "modifier",
  "name": "theme",
  "default": "light",
  "contexts": {
    "light": [],
    "dark": [{ "$ref": "./dark.json" }],
    "high-contrast": [{ "$ref": "./high-contrast.json" }]
  }
}
```

Generates:

```css
:root { /* light */ }
[data-theme="dark"] { /* dark overrides */ }
[data-theme="high-contrast"] { /* high-contrast overrides */ }
```

## Custom selectors: the permutations API

When the auto-generated data attribute selectors aren't enough, use `variables.permutations` in your config to control exactly how each context maps to CSS.

### Media queries

Follow the user's OS preference with zero JavaScript:

```ts title="sugarcube.config.ts"
import { defineConfig } from "@sugarcube-sh/cli";

export default defineConfig({
  resolver: "./tokens.resolver.json",
  variables: {
    permutations: [
      { input: { theme: "light" }, selector: ":root" },
      {
        input: { theme: "dark" },
        selector: ":root",
        atRule: "@media (prefers-color-scheme: dark)"
      },
    ]
  }
});
```

Output:

```css
:root {
  --color-surface: #ffffff;
}

@media (prefers-color-scheme: dark) {
  :root {
    --color-surface: #1a1a1a;
  }
}
```

The `atRule` option works with any CSS at-rule — `prefers-color-scheme`, `prefers-contrast`, `pointer`, etc.

<SiteAside type="note">
  When you define `permutations`, auto-generation is disabled entirely. You must include all contexts you want in the output — including the default. If you omit the light permutation above, only the dark media query block would be generated.
</SiteAside>

### Compound selectors

Selectors can be arrays — they get comma-joined in the output:

```ts
{
  input: { theme: "dark" },
  selector: [
    "[data-color-mode=light][data-light-theme*=dark]",
    "[data-color-mode=dark][data-dark-theme*=dark]"
  ]
}
```

Output:

```css
[data-color-mode=light][data-light-theme*=dark],
[data-color-mode=dark][data-dark-theme*=dark] {
  --color-surface: #1a1a1a;
}
```

### Using --input from the CLI

The `--input` flag is equivalent to defining a single permutation. Use it for one-off builds:

```bash frame="none"
sugarcube generate --input theme=dark --selector "[data-theme='dark']"
```

Without `--selector`, output goes to `:root`:

```bash frame="none"
sugarcube generate --input brand=ocean --variables dist/ocean.css --variables-only
```

<SiteAside type="note">
  When `--input` is used alongside a config that defines `permutations`, the config permutations are ignored. The `--input` flag always takes priority.
</SiteAside>

## Advanced: multiple modifiers and brands

### Multiple modifiers

A resolver can have more than one modifier. A common example is **brand** (palette colors) and **theme** (light/dark surfaces). These are separate concerns that don't overlap:

```json title="tokens.resolver.json"
{
  "version": "2025.10",
  "resolutionOrder": [
    {
      "type": "set",
      "name": "base",
      "sources": [{ "$ref": "./base.json" }]
    },
    {
      "type": "modifier",
      "name": "brand",
      "default": "default",
      "contexts": {
        "default": [],
        "ocean": [{ "$ref": "./ocean.json" }],
        "forest": [{ "$ref": "./forest.json" }]
      }
    },
    {
      "type": "modifier",
      "name": "theme",
      "default": "light",
      "contexts": {
        "light": [],
        "dark": [{ "$ref": "./dark.json" }]
      }
    }
  ]
}
```

Define a permutation per context.

```ts title="sugarcube.config.ts"
import { defineConfig } from "@sugarcube-sh/cli";

export default defineConfig({
  resolver: "./tokens.resolver.json",
  variables: {
    permutations: [
      { input: { brand: "default" }, selector: ":root" },
      { input: { brand: "ocean" }, selector: "[data-brand=\"ocean\"]" },
      { input: { brand: "forest" }, selector: "[data-brand=\"forest\"]" },
      { input: { theme: "dark" }, selector: "[data-theme=\"dark\"]" },
    ]
  }
});
```

Output:

```css
:root { --color-primary: red; --color-surface: white; }
[data-brand="ocean"] { --color-primary: blue; }
[data-brand="forest"] { --color-primary: green; }
[data-theme="dark"] { --color-surface: #1a1a1a; }
```

In HTML, set a data attribute for each active context:

```html
<html data-brand="ocean" data-theme="dark">
  <!-- gets --color-primary: blue from brand, --color-surface: #1a1a1a from theme -->
```

This works because the modifiers are **orthogonal** — they override different tokens, so there's no conflict. The DTCG spec [recommends this approach](https://www.designtokens.org/TR/2025.10/resolver/#orthogonality).

<SiteAside type="tip">
  If two modifiers need to override the same token (e.g., brand-one needs a different dark surface color than brand-two), compose them into one modifier's contexts in the resolver. For example, create `theme` contexts like `dark`, `ocean-dark`, `forest-dark` — each containing the specific overrides that combination needs.
</SiteAside>

### Per-brand output files

Instead of one CSS file with all brands, you can generate a separate file per brand. Use the `path` option:

```ts title="sugarcube.config.ts"
variables: {
  permutations: [
    { input: { brand: "ocean" }, selector: ":root", path: "dist/ocean.css" },
    { input: { brand: "forest" }, selector: ":root", path: "dist/forest.css" },
  ]
}
```

Or from the CLI:

```bash frame="none"
sugarcube generate --input brand=ocean --variables dist/ocean.css --variables-only
sugarcube generate --input brand=forest --variables dist/forest.css --variables-only
sugarcube generate --utilities-only
```

### Resolution order

Modifiers in `resolutionOrder` are applied in sequence. Later entries override earlier ones:

```json
{
  "resolutionOrder": [
    { "type": "set", "name": "base", "sources": [...] },
    { "type": "modifier", "name": "brand", ... },
    { "type": "modifier", "name": "theme", ... }
  ]
}
```

If two modifiers override the same token, the one later in the array wins. Structure your resolver accordingly.
