## General setup

In the root of an existing project, run:

<PackageManagerTabs command="npx @sugarcube-sh/cli init" />

This walks you through setup and installs the CLI locally, so you can use `sugarcube` instead of `npx @sugarcube-sh/cli` going forward.

That's it. 🎉

You can now start using sugarcube.

### Using sugarcube

How you use sugarcube depends on whether you installed the Vite plugin during init.

#### With the Vite plugin

Add the plugin to your build config:

```ts title="vite.config.ts"
import { defineConfig } from "vite";
import sugarcube from "@sugarcube-sh/vite";

export default defineConfig({
    plugins: [sugarcube()],
});
```

Then import the virtual module in your project's entrypoint:

```ts
import "virtual:sugarcube.css";
```

#### Without the Vite plugin

Run the CLI to generate CSS files:

<PackageManagerTabs command="sugarcube generate" />

Then import the output where you import your other styles:

```css
@import "./src/styles/global/tokens.variables.gen.css";
```

<SiteAside type="caution">
    The exact import path will depend on your project structure and any customisation. Check the generated file(s) to see where they are written.
</SiteAside>

<SiteAside type="tip">
    Add a script to your `package.json` for convenience, e.g. `"generate:styles": "sugarcube generate --watch"`
</SiteAside>

### If you added CUBE CSS

If you chose CUBE CSS during `init`, those files need to be imported separately.

**Why?** CUBE CSS files are **source files** — not virtual modules. They're generated once, then they're yours to own and modify. That's why they're not bundled into `virtual:sugarcube.css`.

Import them alongside your other styles:

```css title="styles/index.css"
@import "./global/global.css";
@import "./compositions/flow.css";
@import "./compositions/cluster.css";
/* ... add the files you're using */
```
