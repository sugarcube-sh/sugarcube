<SiteAside type="tip" title="Shorter commands">
    For shorter syntax, install `@sugarcube-sh/cli` as a dev dependency. This provides a `sugarcube` command you can use in npm scripts, e.g.: `"tokens:generate": "sugarcube generate"`
</SiteAside>

## Commands

- **`init`** - Set up a new project
- **`generate`** - Generate CSS variables and utility classes from your design tokens. Supports `--watch` for continuous regeneration during development. Use this if you are *not* using the Vite plugin.
- **`validate`** - Validate design token files for W3C compliance
- **`components`** - Install copy-to-use components for your framework
- **`cube`** - Add CUBE CSS architecture to your project

## Typical Workflow

1. **Initialize**: `npx @sugarcube-sh/cli init` - Sets up starter kit, detects framework, installs dependencies
2. **Generate** CSS (if not using the Vite plugin): `npx @sugarcube-sh/cli generate` - Processes tokens, outputs CSS
3. **Add components** as needed: `npx @sugarcube-sh/cli components`
4. **Validate** tokens: `npx @sugarcube-sh/cli validate`

## Using with UnoCSS CLI (if not using the Vite plugin)

Want utilities like `flex`, `grid`, or `w-full` alongside your token-based utilities? Run the sugarcube CLI alongside the [UnoCSS CLI](https://unocss.dev/integrations/cli):

```bash
# Terminal 1: Token-based CSS
sugarcube generate --watch

# Terminal 2: Preset utilities
unocss "src/**/*.tsx" --watch --preset wind4
```

Then import both:

```css
@import "./styles/global/tokens.variables.gen.css";
@import "./styles/utilities/utilities.gen.css";
@import "./uno.css";
```

Sugarcube handles your design token utilities (`.bg-primary`, `.text-heading`, etc.) while UnoCSS handles preset utilities (`.flex`, `.grid`, `.gap-4`, etc.).

<SiteAside type="note">
    Avoid token names that match preset utility names (like `color.red.500`) to prevent class name conflicts.
</SiteAside>
