Sugarcube generates utility classes from your design tokens. You tell it which CSS properties to create utilities for and which tokens to use.

<SiteAside type="note">
    By default, utilities are only generated if they're actually used in your markup. If a class doesn't appear in your code, it won't be generated. When you need a class that the scanner can't see — one built at runtime, or living in a file it doesn't scan — reach for [`safelist`](#forcing-generation-with-safelist).
</SiteAside>

## The mental model

A utility configuration maps **tokens** to **CSS classes**. Each entry in `utilities.classes` has three parts:

```ts
utilities: {
  classes: {
    "background-color": {   // ← CSS property (what gets applied)
      source: "color.*",    // ← Token pattern (which tokens to use)
      prefix: "bg"          // ← Class prefix (what the class starts with)
    }
  }
}
```

This generates classes like `.bg-primary` that apply `background-color: var(--color-primary)`.

**The formula:**

<Table>
| Part | What it does | Example |
|------|--------------|---------|
| Key | The CSS property applied | `background-color` |
| `source` | Which tokens to match | `color.*` → all tokens under `color` |
| `prefix` | Class name starts with this | `bg` → `.bg-primary`, `.bg-secondary` |
</Table>

## Basic example

Given these tokens:

```json title="colors.json"
{
  "color": {
    "$type": "color",
    "primary": { "$value": "#0066cc" },
    "secondary": { "$value": "#00aa00" }
  }
}
```

This configuration:

```ts title="sugarcube.config.ts"
import { defineConfig } from "@sugarcube-sh/cli";

export default defineConfig({
  utilities: {
    classes: {
      "background-color": {
        source: "color.*",
        prefix: "bg"
      }
    }
  }
});
```

Generates these utilities:

```css
.bg-primary {
  background-color: var(--color-primary);
}

.bg-secondary {
  background-color: var(--color-secondary);
}
```

## Multiple properties from the same tokens

You can create different utility sets from the same tokens by changing the CSS property and prefix:

```ts title="sugarcube.config.ts"
import { defineConfig } from "@sugarcube-sh/cli";

export default defineConfig({
  utilities: {
    classes: {
      // Text colors
      color: {
        source: "color.*",
        prefix: "text"
      },
      // Background colors
      "background-color": {
        source: "color.*",
        prefix: "bg"
      },
      // Border colors
      "border-color": {
        source: "color.*",
        prefix: "border"
      }
    }
  }
});
```

Now `.text-primary`, `.bg-primary`, and `.border-primary` all work, each applying a different CSS property.

## Default prefix behavior

If you omit `prefix`, sugarcube uses the first part of `source` (before the first dot):

```ts
// These are equivalent:
utilities: { classes: { color: { source: "color.*" } } }
utilities: { classes: { color: { source: "color.*", prefix: "color" } } }
```

Both generate `.color-primary`, `.color-secondary`, etc.

<Table>
| `source` | Default prefix | Class names |
|----------|----------------|-------------|
| `color.*` | `color` | `.color-primary` |
| `space.*` | `space` | `.space-sm`, `.space-md` |
| `font.weight.*` | `font` | `.font-bold`, `.font-normal` |
</Table>

## CSS custom properties

You can set CSS custom properties instead of standard properties. This is useful for layout patterns like CUBE CSS:

```ts title="sugarcube.config.ts"
import { defineConfig } from "@sugarcube-sh/cli";

export default defineConfig({
  utilities: {
    classes: {
      "--flow-space": {
        source: "space.*",
        prefix: "flow-space"
      }
    }
  }
});
```

Generates:

```css
.flow-space-sm {
  --flow-space: var(--space-sm);
}

.flow-space-md {
  --flow-space: var(--space-md);
}
```

Your CSS can then reference `--flow-space` to control spacing between elements.

## Nested tokens

When tokens are nested deeper, the full path after the wildcard becomes part of the class name.

Given tokens like:

```json title="colors.json"
{
  "color": {
    "text": {
      "primary": { "$value": "#000" },
      "muted": { "$value": "#666" }
    },
    "fill": {
      "solid": { "$value": "#fff" },
      "soft": { "$value": "#f5f5f5" }
    }
  }
}
```

With `source: "color.*"` and `prefix: "bg"`:

```css
.bg-text-primary { background-color: var(--color-text-primary); }
.bg-text-muted { background-color: var(--color-text-muted); }
.bg-fill-solid { background-color: var(--color-fill-solid); }
.bg-fill-soft { background-color: var(--color-fill-soft); }
```

### Targeting a specific group

To only match tokens in a specific nested path, be more specific in `source`:

```ts
utilities: {
  classes: {
    color: {
      source: "color.text.*",  // Only tokens under color.text
      prefix: "text"
    }
  }
}
```

Generates: `.text-primary`, `.text-muted` (shorter class names, only text colors)

### Avoiding duplicate prefixes with `stripDuplicates`

Sometimes your prefix matches part of the token path, creating awkward names like `.text-text-primary`.

Use `stripDuplicates` to clean this up:

```ts
utilities: {
  classes: {
    color: {
      source: "color.*",
      prefix: "text",
      stripDuplicates: true
    }
  }
}
```

Without `stripDuplicates`: `.text-text-primary`, `.text-text-muted`
With `stripDuplicates`: `.text-primary`, `.text-muted`

## Directional utilities

For spacing properties, generate directional variants:

```ts title="sugarcube.config.ts"
import { defineConfig } from "@sugarcube-sh/cli";

export default defineConfig({
  utilities: {
    classes: {
      padding: {
        source: "space.*",
        prefix: "p",
        directions: ["all", "top", "right", "bottom", "left", "x", "y"]
      }
    }
  }
});
```

With a `space.sm` token, this generates:

<Table>
| Class | CSS Property |
|-------|--------------|
| `p-sm` | `padding` (all sides) |
| `pt-sm` | `padding-block-start` (top) |
| `pr-sm` | `padding-inline-end` (right) |
| `pb-sm` | `padding-block-end` (bottom) |
| `pl-sm` | `padding-inline-start` (left) |
| `px-sm` | `padding-inline` (horizontal) |
| `py-sm` | `padding-block` (vertical) |
</Table>

Note: Directions use CSS logical properties for better internationalization support.

## Forcing generation with `safelist`

By default a utility is only generated if its class appears somewhere the scanner can read. Two common situations slip through that net. The first is a class name your code assembles at runtime, like `bg-${tone}`, which never appears literally in the source for the scanner to find. The second is a class that lives outside the scanned files altogether — CMS content, Markdown, or a third-party script. `safelist` tells sugarcube to generate a utility whether or not it shows up in your markup.

You add it to whichever entry you want to force, alongside `source` and `prefix`.

### Forcing the whole family

Set `safelist: true` to generate a class for every token that `source` matches, exactly as if each one had appeared in your code:

```ts title="sugarcube.config.ts"
utilities: {
  classes: {
    "background-color": {
      source: "color.*",
      prefix: "bg",
      safelist: true
    }
  }
}
```

Every token under `color` now gets a `.bg-*` class, used or not. If the entry also has `directions`, the full directional set is generated for each token as well.

### Forcing a specific subset

Set `safelist` to an array to force only the classes you name.

```ts title="sugarcube.config.ts"
utilities: {
  classes: {
    "background-color": {
      source: "color.*",
      prefix: "bg",
      safelist: ["primary", "danger", "success"]
    }
  }
}
```

That forces `.bg-primary`, `.bg-danger`, and `.bg-success` to be written to your CSS even if they never appear in your markup.

### What to write in the array

Each string is a **segment**, not a full class name and not a token path. Write it the same way you'd write the class in your markup, with the prefix removed. Sugarcube adds the prefix back for you — and when the entry has `directions`, it generates every direction variant from that one segment. In the example above you write `primary`, not `bg-primary` and not `color.primary`.

Three situations change what the segment looks like:

<Table>
| Situation | Tokens / config | Segment you write | Classes it forces |
|-----------|-----------------|-------------------|-------------------|
| Nested token | `color.fill.solid`, prefix `bg` | `fill-solid` | `.bg-fill-solid` |
| `stripDuplicates` | `color.text.muted`, prefix `text` | `muted` | `.text-muted` |
| `directions` | `space.sm`, prefix `p`, `directions: ["all"]` | `sm` | `.p-sm`, `.pt-sm`, `.pr-sm`, `.pb-sm`, `.pl-sm`, `.px-sm`, `.py-sm` |
</Table>

For a nested token, join the path below the wildcard with hyphens, so `color.fill.solid` under `source: "color.*"` becomes `fill-solid`. With `stripDuplicates` turned on, write the shortened form you'd actually type, so `color.text.muted` with prefix `text` is `muted`, not `text-muted`. With `directions`, write only the token part and let sugarcube expand the variants, so `space.sm` is just `sm`.

If a segment doesn't correspond to a real token under `source`, no class is produced for it — a typo costs you nothing worse than a missing utility.

<SiteAside type="note">
    `safelist` works the same way with both the Vite plugin and the CLI, and it doesn't change anything about classes you already use in markup — those are still generated on demand as usual.
</SiteAside>

## Just-in-time compilation

Only utilities actually used in your code are generated — you don't pay for unused utilities. This works automatically with both the Vite plugin and CLI. The one exception is anything you've marked with [`safelist`](#forcing-generation-with-safelist), which is generated regardless of use.

## Using the Vite plugin vs CLI

### With the Vite plugin

Utilities are generated on-the-fly during development and build:

```ts title="vite.config.ts"
import sugarcube from "@sugarcube-sh/vite";

export default {
  plugins: [sugarcube()]
};
```

### Without the plugin (static generation)

Generate utilities as a static CSS file:

<PackageManagerTabs command="sugarcube generate" />

This scans your codebase for used utilities and writes `utilities.gen.css` to your styles directory.

Import it:

```css
@import "./src/styles/utilities/utilities.gen.css";
```

Use `--watch` to automatically regenerate when tokens or markup change:

<PackageManagerTabs command="sugarcube generate --watch" />

## Extending with UnoCSS presets

Add UnoCSS presets for utilities beyond what tokens provide (like `flex`, `grid`, `absolute`):

```ts title="vite.config.ts"
import sugarcube from "@sugarcube-sh/vite";
import presetWind3 from "@unocss/preset-wind3";

export default {
  plugins: [
    sugarcube({
      unoOptions: {
        presets: [presetWind3({ preflight: false })]
      }
    })
  ]
};
```
